WHAT IS FINE-TUNING?

The mental model every beginner must build first. Understand this deeply and everything else falls into place.

1.1 The Big Picture

Imagine you are hiring an employee. Pre-training is like sending that employee to college for 4 years — they learn math, history, science, English, coding — everything. After college they are a "general-purpose intelligent person."

You run a hospital. You need them to know medical terminology, how to read ECG reports, write clinical notes. That specialization process — taking the already-smart person and teaching them your specific domain — is Fine-Tuning.

📌 Core Definition

Fine-Tuning = taking a pre-trained LLM (already knows language) and continuing to train it on your domain-specific data so it learns your task, style, domain vocabulary, or behavior.

Pre-trained LLM: trained on 1 Trillion+ tokens from the entire internet (costs millions $$$)
Fine-Tuned LLM: you spend a few hours/days + a few hundred $ to specialize it for your use case.

1.2 Pre-Training vs Fine-Tuning vs RAG — The Full Picture

This is the MOST important decision you make before writing any code. Fine-tuning is not always the right tool.

ApproachWhat It DoesWhen To UseCost
PromptingAdd instructions to the inputQuick prototyping, general tasks$0
RAGRetrieve relevant context at inference timeFresh knowledge, large knowledge base, frequently updated dataLow
Fine-TuningChange model weights for your taskCustom style, domain vocab, consistent format, latency mattersMedium

1.4 The Mathematics of Fine-Tuning

A pre-trained model has weights θ (theta). These are billions of numbers encoding all knowledge. During pre-training, the model minimized loss L_pretrain on a huge corpus. Those weights are now fixed.

During fine-tuning: we take those weights θ_pretrain and continue gradient descent on our small dataset D_ft:

θ_finetuned = θ_pretrain − α · ∇θ L(θ; D_ft)
WHERE:
α = learning rate (very small, ~1e-4 to 1e-5)
L = your task loss (usually cross-entropy)
∇θ = gradient (direction to update weights)
D_ft = your fine-tuning dataset
KEY INSIGHT: We START from a great point in weight space. We don't train from scratch. That is why fine-tuning needs much less data and compute.

2.2 Attention Mechanism — The Core of LLMs

Attention(Q, K, V) = softmax( Q·Kᵀ / √d_k ) · V
Q = Query matrix (what am I looking for?)
K = Key matrix (what information do I have?)
V = Value matrix (what is the actual content?)
d_k = dimension of keys (scaling factor to prevent vanishing gradients)

LoRA adds ΔW to Q and V projections: W_q_new = W_q + B_q × A_q

2.3 Tokenization — Understanding Tokens

Before text enters the model, it is split into tokens. Tokens are sub-word pieces — not full words. This is critical to understand for dataset preparation.

💡 Real Example

The word "Fine-tuning" becomes tokens: ["▁Fine", "-", "tun", "ing"] (4 tokens, not 1 word!)
Rule of thumb: 1 token ≈ 0.75 words. 1000 tokens ≈ 750 words.
Your max_seq_length=1024 means ~768 words max per training example.

4.3 Full Data Pipeline Code

PYTHON — Complete Data Processing Pipeline
from datasets import load_dataset, Dataset
from transformers import AutoTokenizer
import json

# ── Load from Hugging Face Hub ───────────────────────────
dataset = load_dataset('tatsu-lab/alpaca')  # 52k instruction examples
print(dataset['train'][0])  # See first example

# ── Load from your own JSONL file ───────────────────────
dataset = load_dataset('json', data_files={
    'train': 'my_data.jsonl',
    'test': 'my_test.jsonl'
})

# ── Apply prompt template ────────────────────────────────
ALPACA_TEMPLATE = """### Instruction:\n{instruction}\n\n### Input:\n{input}\n\n### Response:\n{output}"""

def format_prompt(sample):
    return {'text': ALPACA_TEMPLATE.format(
        instruction=sample['instruction'],
        input=sample['input'],
        output=sample['output']
    )}

dataset = dataset.map(format_prompt, remove_columns=dataset['train'].column_names)

# ── Data Quality Checks ─────────────────────────────────
tokenizer = AutoTokenizer.from_pretrained('mistralai/Mistral-7B-v0.1')

def quality_check(example):
    # Remove empty outputs
    if len(example['text'].strip()) < 50:
        return False
    # Check token length (filter too long)
    tokens = tokenizer(example['text'], return_tensors='pt')
    return tokens['input_ids'].shape[1] <= 1024

dataset = dataset.filter(quality_check)

# ── Shuffle and split ────────────────────────────────────
dataset = dataset['train'].shuffle(seed=42)
split = dataset.train_test_split(test_size=0.1, seed=42)
train_data, eval_data = split['train'], split['test']
print(f'Train: {len(train_data)}, Eval: {len(eval_data)}')

4.4 Data Quality Rules — The Non-Negotiables

✅ GOOD DATA PRACTICES
  • Minimum 500 examples for task FT, 5000+ for best results
  • Diverse examples covering all edge cases you care about
  • Consistent formatting — same template every single time
  • Deduplicate — no repeated examples
  • Balanced classes for classification tasks
  • Filter by token length before training
  • Have a human review 100 random samples
❌ COMMON DATA MISTAKES
  • Mixing different prompt templates in same dataset
  • Including examples with empty or very short outputs
  • Not checking for personally identifiable information (PII)
  • Using auto-generated data without quality check
  • Ignoring class imbalance in classification
  • Not splitting train/eval — model memorizes all data
  • Forgetting to add EOS token at end of responses
💡 Industry Secret: Self-Instruct Method

If you don't have labeled data, use GPT-4 to generate it! Write 20 seed examples in your domain → prompt GPT-4 to generate 1000 more → human-filter the best 500 → fine-tune your open-source model.

This is exactly how Alpaca (Stanford), Vicuna, and most open-source instruction models were built. A $100 GPT-4 API bill creates a dataset worth millions of hours of human annotation.

FULL FINE-TUNING (FFT)

The classical approach — update every single parameter. Understand this deeply before PEFT methods make sense.

5.1 What is Full Fine-Tuning?

Every weight in the model gets gradient updates. 7B parameters = 7 billion gradient updates per backward pass. This is powerful but expensive. Memory formula is critical:

Total GPU Memory = Weights + Gradients + Optimizer States + Activations
FP32 training: 4B (weights) + 4B (grad) + 8B (Adam) = 16 bytes/param
FP16 training: 2B (weights) + 2B (grad) + 8B (Adam FP32) = 12 bytes/param

7B model FP32: 7B × 16B = 112 GB ← needs 2× A100 80GB!
7B model FP16: 7B × 12B = 84 GB ← still needs 2× A100
7B QLoRA 4-bit: ~3.5GB weights + ~6GB LoRA = ~10 GB ✅ single GPU!

5.2 Full Fine-Tuning Code

PYTHON — Complete FFT with GPT-2
from transformers import (AutoModelForCausalLM, AutoTokenizer,
                          TrainingArguments, Trainer,
                          DataCollatorForLanguageModeling)
from datasets import load_dataset
import torch

# 1. Load GPT-2 (small enough for free GPU)
model_name = 'gpt2'
tokenizer = AutoTokenizer.from_pretrained(model_name)
tokenizer.pad_token = tokenizer.eos_token  # GPT-2 has no pad token!
model = AutoModelForCausalLM.from_pretrained(model_name)
print(f'Parameters: {sum(p.numel() for p in model.parameters()):,}')

# 2. Load and tokenize dataset
dataset = load_dataset('wikitext', 'wikitext-2-raw-v1')

def tokenize_function(examples):
    return tokenizer(examples['text'], truncation=True, max_length=512, padding='max_length')

tokenized = dataset.map(tokenize_function, batched=True, remove_columns=['text'])

# 3. Data collator for causal LM
data_collator = DataCollatorForLanguageModeling(tokenizer=tokenizer, mlm=False)
# mlm=False → causal LM (predict next token), not BERT-style masked LM

# 4. Training arguments
training_args = TrainingArguments(
    output_dir='./gpt2-finetuned',
    num_train_epochs=3,
    per_device_train_batch_size=4,
    per_device_eval_batch_size=4,
    warmup_steps=100,      # Gradual LR increase (prevents early instability)
    weight_decay=0.01,     # L2 regularization (prevents overfitting)
    learning_rate=2e-5,    # SMALL! Much smaller than pre-training (~1e-4)
    fp16=True,             # Mixed precision: halves memory, speeds up
    logging_steps=50,
    eval_strategy='epoch',
    save_strategy='epoch',
    load_best_model_at_end=True,  # Auto-save best checkpoint
    report_to='wandb',            # Log metrics to Weights & Biases
)

# 5. Trainer and train
trainer = Trainer(
    model=model,
    args=training_args,
    train_dataset=tokenized['train'],
    eval_dataset=tokenized['validation'],
    data_collator=data_collator,
)
trainer.train()
trainer.save_model('./gpt2-finetuned-final')
⚠️ Catastrophic Forgetting

Full Fine-Tuning can cause catastrophic forgetting — the model "forgets" its pre-trained general knowledge when you train too aggressively on a small domain dataset. Mitigation: use a very small learning rate (1e-5), few epochs (1-3), and consider LoRA instead. This is the #1 reason industry prefers PEFT.

PARAMETER-EFFICIENT FINE-TUNING (PEFT)

Train 0.1% of parameters, get 95% of the performance. This is how the entire industry fine-tunes LLMs today.

6.1 The PEFT Revolution

Full FT is impractical for most. PEFT methods freeze the original model weights and only add/train a tiny set of new parameters. The results are nearly identical to full FT at a fraction of the cost.

MethodKey IdeaTrainable ParamsBest For
LoRALow-rank matrices added to attention weights0.1–1%Most tasks — industry default
QLoRALoRA + 4-bit quantized base model0.1–1%Low-VRAM training (10GB GPU)
Prefix TuningTrainable vectors prepended to K, V every layer0.1%When you want model frozen
Prompt TuningTrainable tokens at input only0.01%Smallest possible footprint
Adapter LayersSmall MLP modules inside transformer layers0.5–3%Multi-task fine-tuning
IA3Scale attention activations with learned vectors0.01%Fewest params, fast experiments
AdaLoRADynamic rank allocation across matrices0.1–1%Complex tasks needing flexibility
DoRADecompose weights into magnitude + direction0.1–1%Better than LoRA on some tasks
📌 Why PEFT Wins in Industry

LLaMA-2 7B: LoRA trains ~4M params instead of 7,000M = 0.06% of total!
Training is 3–10× faster, uses 4–8× less GPU memory
You can store 100 task-specific LoRA adapters (each ~20MB) instead of 100 full model copies (each 14GB)
Switching tasks = swapping a small file. Not loading a new 14GB model.

LoRA — LOW-RANK ADAPTATION

The most important fine-tuning method you must master. Every serious ML engineer must understand this deeply — the math, the intuition, every hyperparameter.

7.1 The Core Intuition

LoRA (Hu et al. 2021) is based on one critical observation: when we fine-tune a model on a specific task, the weight changes (ΔW) are low-rank.

This means the change can be expressed as the product of two MUCH smaller matrices:

W_new = W_pretrained + ΔW = W_pretrained + B × A
W_pretrained: shape [d × d] — FROZEN during training
A: shape [r × d] — initialized with random Gaussian, TRAINED
B: shape [d × r] — initialized with ZEROS (so ΔW starts at 0), TRAINED
r = rank, a small number: 4, 8, 16, 32, 64

Forward pass: output = W(x) + (B × A)(x) × (α / r)
α (lora_alpha) = scaling factor, usually set to 2×r

7.2 LoRA Math: How Much Do You Save?

Example: W_q matrix in LLaMA-2 7B = [4096 × 4096]

Full FT trainable: 4096 × 4096 = 16,777,216 params

LoRA with r=16:
  Matrix A: 16 × 4096 = 65,536
  Matrix B: 4096 × 16 = 65,536
  Total: 131,072 params (0.78% of original!)

LLaMA-2 7B has 32 layers, each with Q, K, V, O:
Full FT: 7,000,000,000 trainable params
LoRA r=16 on Q+V: ~4,000,000 trainable params (0.06%!)

7.3 Complete LoRA Fine-Tuning Code

PYTHON — Full LoRA with SFTTrainer
import torch
from transformers import AutoModelForCausalLM, AutoTokenizer
from peft import LoraConfig, get_peft_model, TaskType
from trl import SFTTrainer, SFTConfig
from datasets import load_dataset

# 1. Load tokenizer
MODEL_NAME = 'mistralai/Mistral-7B-v0.1'
tokenizer = AutoTokenizer.from_pretrained(MODEL_NAME, trust_remote_code=True)
tokenizer.pad_token = tokenizer.eos_token
tokenizer.padding_side = 'right'  # Critical: pad on right for causal LM

# 2. Load model in FP16 (no quantization for LoRA)
model = AutoModelForCausalLM.from_pretrained(
    MODEL_NAME,
    torch_dtype=torch.float16,
    device_map='auto',  # auto-spread across available GPUs
)
model.config.use_cache = False       # Disable KV cache during training
model.config.pretraining_tp = 1       # Disable tensor parallelism

# 3. LoRA Config — the most important part
lora_config = LoraConfig(
    r=16,                # Rank: 8=less params, 64=more quality
    lora_alpha=32,       # Scaling: always set to 2×r
    target_modules=[    # Which layers get LoRA adapters
        'q_proj', 'k_proj',  # Query, Key attention
        'v_proj', 'o_proj',  # Value, Output attention
        'gate_proj', 'up_proj', 'down_proj'  # FFN layers
    ],
    lora_dropout=0.05,  # Dropout on LoRA layers (regularization)
    bias='none',        # Don't train biases
    task_type=TaskType.CAUSAL_LM,
)

# 4. Apply LoRA and check trainable params
model = get_peft_model(model, lora_config)
model.print_trainable_parameters()
# Output: trainable params: 3,407,872 || all params: 7,244,300,288 || trainable%: 0.047

# 5. Load and format dataset
dataset = load_dataset('tatsu-lab/alpaca', split='train[:5000]')

def format_alpaca(sample):
    if sample['input']:
        return {'text': f"### Instruction:\n{sample['instruction']}\n\n### Input:\n{sample['input']}\n\n### Response:\n{sample['output']}"}
    return {'text': f"### Instruction:\n{sample['instruction']}\n\n### Response:\n{sample['output']}"}

dataset = dataset.map(format_alpaca)

# 6. Training config
sft_config = SFTConfig(
    output_dir='./mistral-lora',
    num_train_epochs=3,
    per_device_train_batch_size=2,
    gradient_accumulation_steps=4,  # Effective batch = 2×4 = 8
    learning_rate=2e-4,             # Higher than FFT — LoRA handles it
    fp16=True,
    warmup_ratio=0.03,
    lr_scheduler_type='cosine',      # Cosine decay — industry standard
    max_seq_length=1024,
    dataset_text_field='text',
    packing=True,                     # Pack short examples together — 2× speedup
    report_to='wandb',
)

# 7. Train!
trainer = SFTTrainer(model=model, train_dataset=dataset, peft_config=lora_config,
                     args=sft_config, tokenizer=tokenizer)
trainer.train()
trainer.model.save_pretrained('./mistral-lora-adapter')  # Saves only ~20MB!

# 8. Inference: load base + adapter
from peft import PeftModel
base = AutoModelForCausalLM.from_pretrained(MODEL_NAME, torch_dtype=torch.float16)
model_lora = PeftModel.from_pretrained(base, './mistral-lora-adapter')
model_lora = model_lora.merge_and_unload()  # Optional: merge for faster inference

QLoRA — MEMORY MAGIC

Fine-tune a 70B model on a single GPU. QLoRA (2023) changed the game by combining 4-bit quantization with LoRA. This is what most students and startups use.

8.1 The Three QLoRA Innovations

🔢
NF4 — 4-bit NormalFloat
A new data type that stores weights in 4 bits (instead of 16/32) with minimal quality loss. The quantization points match a normal distribution — same distribution as LLM weights.
KEY INNOVATION 1
🔁
Double Quantization
Quantize the quantization constants themselves! Saves an extra ~0.5 bits/param. Sounds like a trick but it works cleanly and adds up to meaningful memory savings.
KEY INNOVATION 2
💾
Paged Optimizers
Use CPU RAM as overflow for optimizer states during gradient spikes. Prevents CUDA out-of-memory errors without crashing training mid-way through an epoch.
KEY INNOVATION 3
Memory comparison for 7B model:

Full FP16: 14 GB weights only
LoRA FP16: 14 GB weights + 0.5 GB LoRA = 14.5 GB
QLoRA NF4: 3.5 GB weights + 0.5 GB LoRA = ~4 GB! ✅

QLoRA + gradient checkpointing = fine-tune 7B on a 10GB GPU!

8.2 Complete QLoRA Code

PYTHON — QLoRA with BitsAndBytes
import torch
from transformers import AutoModelForCausalLM, AutoTokenizer, BitsAndBytesConfig
from peft import LoraConfig, get_peft_model, prepare_model_for_kbit_training

# ── KEY DIFFERENCE: 4-bit BitsAndBytes config ────────────────────
bnb_config = BitsAndBytesConfig(
    load_in_4bit=True,                         # Load model in 4-bit
    bnb_4bit_quant_type='nf4',                  # NormalFloat4 data type
    bnb_4bit_compute_dtype=torch.bfloat16,       # Compute in BF16 (more stable than FP16)
    bnb_4bit_use_double_quant=True,              # Double quantization
)

# Load 4-bit model
model = AutoModelForCausalLM.from_pretrained(
    'meta-llama/Llama-2-7b-hf',
    quantization_config=bnb_config,
    device_map='auto',
)

# ── CRITICAL STEP: prepare for k-bit training ─────────────────────
# This enables gradient checkpointing + casts LayerNorm to FP32
model = prepare_model_for_kbit_training(model)

# LoRA config (same as before, but r higher since model is quantized)
lora_config = LoraConfig(
    r=64,  # Higher rank because quantization reduces expressiveness
    lora_alpha=16,
    target_modules=['q_proj', 'k_proj', 'v_proj', 'o_proj',
                    'gate_proj', 'up_proj', 'down_proj'],
    lora_dropout=0.05, bias='none', task_type='CAUSAL_LM'
)

# Training config with QLoRA-specific settings
sft_config = SFTConfig(
    output_dir='./llama2-qlora',
    num_train_epochs=2,
    per_device_train_batch_size=1,       # Keep small for memory
    gradient_accumulation_steps=8,      # Effective batch = 8
    gradient_checkpointing=True,         # CRITICAL: trades compute for memory
    optim='paged_adamw_32bit',          # Paged optimizer (no OOM spikes)
    learning_rate=2e-4,
    bf16=True,                           # BF16 more stable than FP16 for LLMs
    max_grad_norm=0.3,                  # Gradient clipping (prevents explosions)
    warmup_ratio=0.03,
    lr_scheduler_type='constant',       # Constant LR after warmup
    max_seq_length=2048,
)

tokenizer = AutoTokenizer.from_pretrained('meta-llama/Llama-2-7b-hf')
tokenizer.pad_token = tokenizer.eos_token
from datasets import load_dataset
dataset = load_dataset('tatsu-lab/alpaca', split='train')
dataset = dataset.map(lambda x: {'text': f"### Instruction:\n{x['instruction']}\n\n### Response:\n{x['output']}"})

trainer.train()
trainer.model.save_pretrained('./llama2-qlora-final')
💡 When to use BF16 vs FP16

BF16 (BrainFloat16): Same range as FP32, less precision. Better for LLM training — less prone to overflow. Use when your GPU supports it (A100, H100, RTX 3090+).
FP16 (Half precision): Smaller range, can overflow with large gradients. Use on older GPUs (T4, V100). Add max_grad_norm=0.3 to prevent gradient explosions in FP16.

PREFIX & PROMPT TUNING

Soft prompts that the model learns automatically. Keep the entire base model frozen — only train a few thousand parameters.

9.1 Prompt Tuning

Prepend learnable token embeddings (soft prompts) to the input. The model is completely frozen. Only these embeddings are trained — just 20 virtual tokens = ~20,000 parameters vs billions.

HOW IT WORKS
  • Hard prompt: "Classify as positive/negative:" (fixed text)
  • Soft prompt: 20 learnable vectors (gradient-optimized)
  • These vectors are NOT human-readable
  • They encode task-specific "instructions" in embedding space
  • Works well when you have >100k examples
LIMITATIONS
  • Underperforms LoRA on complex tasks
  • Needs large models (>1B params) to work well
  • Can't match full fine-tuning quality
  • Hard to debug (can't read the soft prompt)
  • Sensitive to initialization
PYTHON — Prompt Tuning & Prefix Tuning
from peft import PromptTuningConfig, PromptTuningInit, get_peft_model

ADAPTER LAYERS

Insert small neural networks inside the Transformer. Popular for multi-task learning where you need one backbone, many task heads.

10.1 Adapter Architecture

Adapters insert small bottleneck MLP modules between/after transformer layers. The original model is frozen; only the adapters train.

Adapter(x) = x + Linear(r→d)(Activation(Linear(d→r)(LayerNorm(x))))
d = model hidden dim (e.g. 4096 for 7B model)
r = bottleneck dim (e.g. 64 — much smaller than d)
The residual connection (+x) ensures identity at initialization
PYTHON — AdaLoRA (Adaptive LoRA)
from peft import AdaLoraConfig, get_peft_model

# AdaLoRA — dynamically allocates rank budget across matrices
# Learns: which weight matrices actually NEED more rank?

    init_r=12,         # Initial rank for all matrices
    target_r=8,        # Final target rank after pruning unimportant singular values
    tinit=200,         # Steps before rank scheduling starts

    deltaT=10,         # Rank update interval (every 10 steps)
    lora_alpha=32,
    lora_dropout=0.1,
    target_modules=['q_proj', 'v_proj'],
    task_type='CAUSAL_LM'

RLHF & DPO

Teaching models human preferences. This is how ChatGPT and Claude were aligned. DPO simplifies this dramatically — same results, no separate reward model needed.

12.1 RLHF — The 3-Stage Pipeline

1
SUPERVISED FINE-TUNING (SFT)

Train the base model on high-quality demonstration data. This creates the SFT model — a good starting point with basic instruction following. Essentially Chapter 11.

Already know this → Chapter 11
2
REWARD MODEL TRAINING

Show humans pairs of responses (A vs B). They pick which is better. Train a separate "reward model" to predict human preference scores. This model outputs: "Response A is 0.87, Response B is 0.42."

Most expensive step — requires human annotators
3
PPO REINFORCEMENT LEARNING

Use the reward model as a reward signal. The LLM generates responses, reward model scores them, PPO optimizer updates LLM weights to generate higher-scoring responses. A KL penalty prevents the model from drifting too far from the SFT model.

Complex to implement — use DPO instead (below)

12.2 DPO — The Game Changer

DPO (Rafailov et al. 2023) eliminates the need for a separate reward model AND PPO. It directly optimizes from preference pairs using a clever mathematical insight: the optimal policy implicitly defines a reward function.

📌 DPO Dataset Format

Each training example has three fields:
prompt: "Explain quantum computing"
chosen: "Quantum computing uses qubits that can be 0 and 1 simultaneously..." (good response)
rejected: "Quantum computing is very complicated." (bad response)

DPO trains the model to assign higher probability to chosen responses. The reference model (copy of SFT model) acts as a KL constraint to prevent the model from drifting too far.

PYTHON — Complete DPO Training

from evaluate import load
import torch

# ── 1. Perplexity ─────────────────────────────────────────────
def calculate_perplexity(model, tokenizer, texts, device='cuda'):
    model.eval()
    total_loss, total_tokens = 0, 0
    with torch.no_grad():
        for text in texts:
            inputs = tokenizer(text, return_tensors='pt').to(device)
            outputs = model(**inputs, labels=inputs['input_ids'])
            total_loss += outputs.loss.item() * inputs['input_ids'].shape[1]
            total_tokens += inputs['input_ids'].shape[1]
    return torch.exp(torch.tensor(total_loss / total_tokens)).item()

# ── 2. ROUGE Score (for summarization) ───────────────────────
rouge = load('rouge')
results = rouge.compute(
    predictions=['The model predicted this summary'],
    references=['The actual reference summary here']
)
print(results)  # {'rouge1': 0.5, 'rouge2': 0.2, 'rougeL': 0.4}

⚠️ Watch Your Loss Curves

Healthy training: Train loss ↓ steadily. Val loss ↓ then levels off.
Overfitting: Train loss ↓ but val loss starts ↑ — stop training here!
Underfitting: Both losses still ↓ at end — train more epochs or increase LR.
Divergence: Loss suddenly → NaN or goes ↑ — LR too high or corrupted data.
